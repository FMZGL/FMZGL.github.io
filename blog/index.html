AI新名词、新热点

2026-07-15

一、什么是Harness Engineering？

随着大模型能力趋于同质化，AI 工业界正在从“提示词工程 (Prompt Engineering)”全面迈向“智能体牵引工程 (Harness Engineering)”。

业界最新公式：

Agent（智能体） = Model（大模型） + Harness（控制脚手架）

如果把强大的大模型比作“马力强劲但易失控的 F1 发动机”，那么 Harness Engineering 就是在设计底盘、方向盘、刹车和防滚架，从而确保大模型能安全、精准地完成复杂任务。

1. 多层级防护与防幻觉架构 (Guardrails & Constraints)

不再依赖大模型的自我纠错，而是从外部构建硬性约束。

  • **运行时拦截 (Runtime Hooks)**：在代码或指令执行前放入沙箱测试，报错则强制打回重写。
  • 输出校验关卡：强制约束模型输出格式（如严格的 JSON schema），否则不予放行。

2. 工具调用的标准化 (Tool Use & MCP)

赋予模型稳定、标准化的“手和脚”。

  • **MCP (模型上下文协议)**：通过统一的标准协议，将企业内部的数据库、API、Git 仓库等外部环境无缝接入。
  • 动态工具路由：设计精密的调用逻辑，使智能体能自主决定工具的调用顺序与组合策略。

3. 多智能体协作网络 (Multi-Agent Orchestration)

从“单兵作战”向“工厂流水线”升级。

  • 流程式架构：将复杂任务拆解，分别分配给专门的代码编写 Agent、测试 Agent 和审核 Agent。
  • Harness 工程师的职责是设计这些 Agent 之间的通信协议、反驳机制以及共识达成流程。

4. 记忆与状态管理 (Memory & State Management)

解决大模型的“无状态”痛点，构建持久化运行能力。

  • 结合向量数据库（如 FAISS、Milvus）和图数据库，为智能体搭建长期记忆库和短期工作区，确保在执行数周的长周期项目时保持上下文连贯。

[!TIP]
总结
Harness Engineering 的核心意义在于：将大模型的不确定性（幻觉与随机性）转化为工业级应用所需的确定性（高可靠与可落地）。这已经是当前 AI 开发与落地过程中的最核心技能。

Harness Engineering 本质是将 AI 的强能力融入传统软件工程的严谨性中。工程师不需要“炼丹（预训练）”，但必须是极强的“系统集成者”与“防御规则制定者”。

以下是该岗位必备的五大核心基础：

1. 软件工程与架构底座 (Core Engineering)

大模型具有不可预测性，Harness 的首要任务就是用严密的架构将其“锁入安全域”。

  • 主流语言：精通 Python 和 TypeScript。
  • 系统集成：具备极强的后端开发能力，精通 RESTful API / GraphQL 的设计与调用。
  • 沙箱隔离：熟练掌握 Docker 与虚拟机环境配置，保障 Agent 生成或执行的外部代码不会危害宿主机。

2. 智能体开发框架 (Agentic Frameworks)

避免重复造轮子，必须熟练驾驭业界前沿框架。

  • 工作流编排：精通 LangGraph、AutoGen 或 CrewAI，能够设计复杂的多智能体协作图（DAG）、状态机及循环反馈链路。
  • 生态集成：熟练使用 LangChain、LlamaIndex，能快速将企业私有数据库和外部工具封装为 Agent 可用的 Tool。

3. 大模型运行机制理解 (LLM Mechanics)

不需要精通底层算法，但必须懂模型的“脾气”与边界。

  • 上下文与记忆：深刻理解 Context Window、Token 计算规则以及多轮对话中的记忆修剪策略。
  • 结构化约束：熟练运用 System Prompt、Function Calling 和 Output Parsers，强制模型 100% 输出符合业务要求的 JSON 数据。
  • 超参调优：精准理解 Temperature、Top-p 等参数对抑制“幻觉”的作用。

4. 防护工程与自动化测试 (Guardrails & Testing)

Harness Engineering 最具技术壁垒的环节——防御性系统设计。

  • 防御性编程：习惯预判异常，熟练编写 Try-Catch 及带退避算法的重试机制（Retry Mechanism）。
  • 护栏组件库：熟悉并应用 NeMo Guardrails 或 Guardrails AI，在输入端过滤恶意指令，在输出端拦截违规内容。
  • 基于契约的测试：能够为大模型充满随机性的输出设计自动化验收脚本。

5. 记忆引擎与向量检索 (Memory & Storage)

为智能体配备“长期记忆外挂”。

  • 向量数据库：精通 FAISS、Milvus 或 Pinecone 的部署与调优，支撑复杂的 RAG 架构。
  • 知识表示：了解图数据库（如 Neo4j），以处理对准确性和逻辑关联度要求极高的业务场景。

核心范式转变：
传统的后端开发是“面向确定性编程”；而 Harness Engineering 是“面向不确定性编程”。不仅要接收大模型给出的正确答案，更要有能力拦截并自动纠正它产生的荒谬输出。

二、什么是世界模型？

在人工智能领域，世界模型（World Model）被公认为是通往通用人工智能（AGI）最核心的钥匙。

传统的语言大模型（如 ChatGPT）本质上是“文字统计接龙”，它们不具备对物理世界的实际认知。而世界模型旨在打破这一限制，让 AI 真正理解并模拟我们所在的物理世界。

1.核心定义

世界模型是一个能够理解物理规则与常识，并能根据当前的动作预测未来状态的 AI 系统。

通俗比喻：
当人类闭上眼睛想象把生鸡蛋砸向墙壁时，脑海中能浮现出鸡蛋碎裂、蛋液溅出的画面。这种在脑海中进行“未来演练”的能力，是因为我们的大脑中有一个懂重力、懂碰撞的“世界模型”。AI 的世界模型也是要赋予机器同样的“脑内沙盘推演”能力。

2.世界模型的三大核心能力

  1. 物理规律直觉 (Intuitive Physics)
    不需要写入严谨的代码公式（如 $F=ma$），模型通过观察就能直觉地理解重力、遮挡、透视、流体动力学等客观规律。

  2. 因果与时序推理 (Causality & Temporal Dynamics)
    不仅知道“画面里有什么”，更知道“接下来会发生什么”。例如，看到风吹能推测树叶会动，并能精准预测环境在一秒钟或一分钟后的状态演变。

  3. 反事实推理 (Counterfactual Reasoning)
    最高级的认知能力。模型能在内部沙盘做假设推演：“如果刚才那辆车没有刹车，现在会发生什么？”

3.现实中的代表性案例

虽然完美的完全体世界模型尚未诞生，但工业界已有几个极具震撼力的雏形：

  • Sora (OpenAI)
    表面上是视频生成工具，实际上被官方定义为“世界模拟器”。Sora 之所以能生成逼真的一镜到底视频，是因为它在海量训练中隐式地学会了 3D 空间结构和摄像机运镜的物理透视法则。

  • 自动驾驶 (如 Tesla FSD v12)
    汽车在高速行驶时，AI 大脑不仅在识别前方车辆，还在实时预测周围所有车辆和行人未来 3 秒内的运动轨迹。这种“脑内预演”系统即是自动驾驶的世界模型。

  • 具身智能机器人的“模拟大脑”
    当机器人跨越障碍物时，其世界模型会在 0.01 秒内在虚拟空间将动作试演一遍，确认不会摔倒后，才下发实际的物理执行指令。

总结

如果说过去的大语言模型是“熟读唐诗三百首”的文科生，那么世界模型就是要培养一个“睁眼看世界、懂物理法则、能预知行动后果”的理科生兼实干家。谁能率先攻克世界模型，谁就能让人工智能在真实的物理世界（如自动驾驶、智能制造）中完美落地。

算法与数据结构模板与经典例题五：树

2026-05-28

前缀、中缀、后缀表达式常见转换方法总结

在计算机科学中，表达式有三种常见的表示方式。它们的主要区别在于运算符（Operator）相对于操作数（Operand）的位置：

  1. 中缀表达式 (Infix Expression)：运算符在两个操作数中间（例如：a + b）。这是人类最习惯的表示法，但需要括号和优先级规则来消除歧义。
  2. 前缀表达式 (Prefix Expression / 波兰式 Polish Notation)：运算符在操作数前面（例如：+ a b）。不需要括号，无歧义，便于计算机处理。
  3. 后缀表达式 (Postfix Expression / 逆波兰式 Reverse Polish Notation, RPN)：运算符在操作数后面（例如：a b +）。不需要括号，无歧义，是计算机（尤其是利用栈结构）评估和计算表达式最常用的格式。

一、 通用万能方法：加括号法（最适合手工转换）

这是人类手动转换最快、最不容易出错的方法。无论进行什么转换，核心思想都是：“利用运算符优先级，把所有子表达式用括号括起来，然后移动运算符。”

核心步骤：

  1. 完全括号化：根据运算符的优先级和结合性，给表达式的所有运算步骤都加上括号。
  2. 移动运算符：
    • 转前缀：将每个括号内的运算符移到该括号的左括号前面。
    • 转后缀：将每个括号内的运算符移到该括号的右括号后面。
  3. 消除括号：擦除所有的左右括号，即得到目标表达式。

📌 实例演练：将 a + b * c - (d / e) 进行转换

步骤 1：完全括号化

按照优先级（先乘除、后加减、括号最先），逐步加上括号：

  1. 先计算 b * c 和 d / e：a + (b * c) - (d / e)
  2. 再计算 a 与 (b * c) 的加法：((a + (b * c)) - (d / e))
  3. 得到完全括号化的表达式：((a + (b * c)) - (d / e))

步骤 2 & 3：移动并消除括号

转换类型 移动过程演示 消除括号后的最终结果
中缀 → 前缀 将运算符移到对应左括号左边：
-(+(a *(b c)) /(d e))		 - + a * b c / d e
中缀 → 后缀 将运算符移到对应右括号右边：
((a (b c)*)+ (d e)/)-		 a b c * + d e / -

二、 图像法：表达式树法（最直观）

表达式树（Expression Tree）是一种二叉树，其中叶子节点代表操作数，非叶节点（内部节点）代表运算符。构建出表达式树后，通过不同的树遍历方式，可以直接输出对应的表达式。

核心步骤：

  1. 构建二叉树：根据优先级将中缀表达式建成一棵二叉树（根节点是最后执行的运算符）。
  2. 遍历二叉树：
    • 前序遍历 (根 → 左 → 右) → 前缀表达式
    • 中序遍历 (左 → 根 → 右) → 中缀表达式 (需还原括号)
    • 后序遍历 (左 → 右 → 根) → 后缀表达式

📌 实例：a + b * c - d / e 对应的树结构

  • 前序遍历 (中-左-right/中-左-右)：从根 - 开始 → - → + → a → * → b → c → / → d → e → - + a * b c / d e
  • 后序遍历 (左-右-中)：从最左下开始 → a → b → c → * → + → d → e → / → - → a b c * + d e / -

三、 计算机算法：栈式转换法（最适合编程实现）

在计算机程序中，我们通常使用栈（Stack）这一数据结构来实现它们之间的互相转换。

1. 中缀表达式 → 后缀表达式 (经典的 Shunting-yard 调度场算法)

这是最著名的算法，由迪杰斯特拉（Dijkstra）发明。

算法逻辑：

准备一个运算符栈和一个输出队列/列表。从左到右扫描中缀表达式的每个字符：

  • 如果是操作数：直接输出（加入到输出队列）。
  • 如果是左括号 (：直接压入运算符栈。
  • 如果是右括号 )：不断弹出运算符栈顶的运算符并输出，直到遇到左括号 (。最后弹出并丢弃左括号 (。
  • 如果是运算符 (如 +, -, *, /)：
    • 比较当前运算符与栈顶运算符的优先级。
    • 如果栈顶运算符优先级 ≥ 当前运算符 (且栈顶不为 ()，则弹出栈顶运算符并输出。重复此步骤，直到栈顶运算符优先级更低或栈为空。
    • 然后将当前运算符压入栈中。
  • 扫描结束后：如果栈中还有运算符，依次弹出并输出。

2. 中缀表达式 → 前缀表达式

算法逻辑（巧妙的逆向法）：

中缀转前缀可以通过“反转-转后缀-再反转”的方式，复用中缀转后缀的逻辑：

  1. 反转中缀表达式的字符串。在反转时，将所有的 ( 换成 )，所有的 ) 换成 (。
  2. 使用修改后的中缀转后缀算法，计算出反转后表达式的后缀形式。

    注：修改点在于，在遇到同等优先级的运算符时，原算法是“栈顶 ≥ 当前”则出栈（左结合），这里为了保持前缀表达式的右结合性质，改为“栈顶 > 当前”才出栈（即同级时不弹栈）。

  3. 反转步骤 2 得到的后缀表达式，得到的结果即为前缀表达式。

3. 后缀表达式 → 中缀表达式

算法逻辑：

使用一个操作数栈。从左到右扫描后缀表达式：

  • 如果是操作数：压入栈中。
  • 如果是运算符：
    1. 从栈中弹出两个操作数：先弹出的作为右操作数 Op_2，后弹出的作为左操作数 Op_1。
    2. 将它们与当前运算符组合成一个带括号的新字符串："( " + Op_1 + " " + Operator + " " + Op_2 + " )"。
    3. 将这个组合后的新字符串重新压入栈中。
  • 扫描结束后：栈顶剩下的唯一元素就是完整的中缀表达式。

4. 前缀表达式 → 中缀表达式

算法逻辑：

与后缀转中缀非常相似，区别在于扫描方向和操作数组装顺序。
从右到左扫描前缀表达式：

  • 如果是操作数：压入栈中。
  • 如果是运算符：
    1. 从栈中弹出两个操作数：先弹出的作为左操作数 Op_1，后弹出的作为右操作数 Op_2。
    2. 将它们组合成："( " + Op_1 + " " + Operator + " " + Op_2 + " )"。
    3. 压入栈中。
  • 扫描结束后：栈顶剩下的唯一元素就是中缀表达式。

5. 前缀与后缀之间的直接转换 (前缀 ↔ 后缀)

如果不需要经过中缀，前缀和后缀之间也可以通过栈直接转换：

转换类型 扫描方向 操作数处理 遇到运算符时的操作
前缀 → 后缀 从右到左 遇到操作数压入栈 弹出两个元素 Op_1 和 Op_2。
组合为：Op_1 + Op_2 + Operator
重新压入栈。
后缀 → 前缀 从左到右 遇到操作数压入栈 弹出两个元素（先弹出的为 Op_2，后弹出的为 Op_1）。
组合为：Operator + Op_1 + Op_2
重新压入栈。

四、 总结与快速对比

下表总结了计算机内部在处理这三类转换时的规律：

转换方向 扫描方向 栈的类型 核心操作特性
中缀 → 后缀 从左到右 运算符栈 比较优先级，根据优先级高低决定运算符出栈或入栈。
中缀 → 前缀 从右到左 运算符栈 类似于反向中缀转后缀，括号方向对调，保持右结合。
后缀 → 中/前 从左到右 操作数栈 每次遇到运算符，弹出两个元素，拼接后重新入栈。
前缀 → 中/后 从右到左 操作数栈 每次遇到运算符，弹出两个元素，拼接后重新入栈。

[!TIP]
记忆小窍门：

  • 涉及中缀到其他的转换，栈里存的是运算符，需要处理优先级。
  • 涉及非中缀到其他（前/后缀相互转或转中缀），栈里存的是操作数（或子表达式字符串），无需考虑优先级。
  • 只要表达式以前缀开头或需要转成前缀，通常从右向左扫描效率最高；反之，若涉及后缀，则从左向右扫描。

大语言模型（LLM）微调方法全景指南

2026-05-21

在实际应用中，大语言模型的微调（Fine-Tuning）是一个多维度的系统工程。我们通常从参数更新策略（怎么改权重）、任务与架构适配（解决什么任务）以及人类偏好对齐（如何符合人类预期）三个维度来理解和选择微调方法。

维度一：参数更新策略（How to update parameters）

参数更新策略决定了在微调过程中，模型的哪些权重需要被训练，哪些需要被冻结。

1. 全参数微调（Full Fine-Tuning, FFT）

  • 原理：更新模型内的所有权重参数。
  • 优缺点：
    • 优点：模型调整最彻底，理论性能上限最高，适合学习全新的领域知识或深刻改变模型的行为模式。
    • 缺点：算力消耗极高（需要数倍于模型体量的高性能显存来存储梯度和优化器状态）；极易引发灾难性遗忘（Catastrophic Forgetting）。

2. 参数高效微调（Parameter-Efficient Fine-Tuning, PEFT）

通过冻结大模型的大部分参数，仅引入并训练极少量的额外参数（通常 <1%），在极低资源下实现接近全参微调的效果。

A. 旁路低秩矩阵类（目前最主流）

  • LoRA (Low-Rank Adaptation - 低秩适应)
    • 原理：冻结原始权重 W，在矩阵乘法旁路并联两个低秩矩阵 A（降维）和 B（升维）。训练时只更新 A 和 B 的参数。
    • 优点：训练完后可将旁路权重无缝合并回主权重，不增加推理延迟。
  • QLoRA (Quantized LoRA)
    • 原理：将基础模型量化为极低精度的 4-bit（NF4格式），在此基础上挂载 LoRA 进行微调。
    • 优点：将显存开销压缩到极致，使得在单张消费级显卡（如 RTX 4090）上微调 33B/70B 级别大模型成为可能。
  • DoRA (Weight-Decomposed Low-Rank Adaptation)
    • 原理：将权重分解为大小（Magnitude）和方向（Direction）两个通道，仅对方向通道应用 LoRA。
    • 优点：拟合能力更逼近全参数微调。

B. 前缀与提示词微调类（修改输入端）

  • Prompt Tuning：在输入的 Embedding 层前拼接一串可学习的连续向量（虚拟 Token），微调时只训练这几个虚拟 Token。
  • Prefix Tuning：在 Transformer 的所有隐藏层（Hidden States）前都插入可训练的前缀虚拟 Token。

C. 插件与选择性微调类

  • Adapter：在 Transformer 的 Self-Attention 和 MLP 层后插入小型瓶颈结构（Adapter 块），微调时仅训练 Adapter。
  • BitFit：仅微调模型中的 Bias（偏置）项，极其轻量但调节能力非常有限。

维度二：任务与架构适配（What task structure to tune for）

任务与架构适配决定了如何改造模型结构、设计输入输出格式，以解决具体的下游业务问题。

1. 序列分类任务（Sequence Classification）

  • 代表任务：情感分析、垃圾邮件分类、意图识别、文本匹配。
  • 架构改造：
    • 在大模型的顶层拼接一个随机初始化的线性映射层（Classification Head）。
    • 提取整句话的向量表示（如 BERT 的 [CLS] 向量，或 LLaMA 最后一个 Token 的输出向量）送入分类头进行 N 分类。
  • 具体微调形式：
    • 线性探测（Linear Probing）：完全冻结模型本体，只训练分类头。适合样本极少、基础模型极强的场景。
    • 全参分类微调：同时训练模型本体和分类头。
    • PEFT分类微调：冻结模型本体，同时训练挂载的 LoRA 参数和分类头（工业界最常用）。

2. 序列标记任务（Token Classification）

  • 代表任务：命名实体识别（NER）、词性标注。
  • 架构改造：为每一个 Token 的输出向量都连接一个分类头，判断每个 Token 的具体标签（如人名、地名等）。

3. 因果语言模型生成任务（Causal LM Generation）

  • 代表任务：智能对话、代码生成、摘要生成。
  • 架构改造：无需修改模型架构，直接使用自回归的方式，预测下一个 Token 的概率分布。

维度三：人类偏好对齐（Alignment）

偏好对齐通常是微调的最后一步，目的是让模型的输出更符合人类的价值观、安全规范以及特定指令风格。

  • RLHF (基于人类反馈的强化学习)：
    • 机制：通过“策略模型 + 奖励模型（Reward Model） + 价值模型 + PPO算法”协同训练。
    • 评价：效果上限极高，但涉及多个模型交互，训练极其不稳定且调参难度极大。
  • DPO (直接偏好优化)：
    • 机制：省去显式的“奖励模型”，直接将损失函数与人类偏好配对数据（Chosen vs Rejected）挂钩，通过简单的二分类交叉熵直接优化大模型。
    • 评价：目前最主流的对齐手段，训练稳定且高效。
  • ORPO (优势比偏好优化)：
    • 机制：将监督微调（SFT）与偏好对齐融为一步，通过优势比损失对负面样本进行惩罚。

总结：选型决策矩阵

微调目标 硬件算力 推荐方案组合 理由
学习全新且庞大的垂直领域知识 极其充沛 FFT (全参数微调) + Causal LM 需要对底层权重进行深度塑造，注入行业壁垒知识。
日常垂直业务任务（如客服对话、写文案） 中等 / 单卡 LoRA + Causal LM 性价比最高，不增加推理延迟，训练极快。
小样本下的文本分类/情感分析 极少 Linear Probing (仅微调分类头) 防止过拟合，保留预训练模型的泛化特征。
大规模业务文本分类/文本匹配 中等 / 单卡 LoRA + Sequence Classification Head 兼顾语义空间的微调与分类头的优化，准确率高。
微调后模型对齐安全规范与人类偏好 中等 DPO 比 RLHF 稳定且省资源，对齐效果优秀。

JAVA与C/C++的内存分配机制

2026-05-18

这次我们抛开所有难懂的专业术语，不用“深拷贝”、“解引用”或者“生命周期管理”这些高大上的词。我们用最接地气的两样东西来打比方：**“口袋（栈）”**和**“大仓库（堆）”**。

我们可以把 Java 和 C++ 的设计思路，简化为两个完全不同的“生活习惯”。

1. Java 的设计思路：严格、死板，但绝不出错

Java 的核心思想是：“程序员容易粗心，所以必须立死规矩。”

为了不让程序崩溃，Java 把所有的数据分成了两个绝对的阵营，谁也不能越界：

规矩一：小东西放“口袋”，大东西放“仓库”

  • 口袋（栈）：只用来放最简单的数字和布尔值（比如 int a = 10;），或者是各种遥控器。
  • 大仓库（堆）：只要是稍微复杂一点的东西（比如一段文字 String、一个链表节点 Node、甚至是一个最普通的整数数组 int[]），统统必须塞进大仓库里。

规矩二：看管大仓库，全靠“遥控器（引用）”

在 Java 里，你根本没办法直接用手摸到仓库里的东西。你手里拿到的，永远是一个“遥控器”。

  • 你写 Node el = new Node();，实际上是在大仓库里建了一个节点，然后把控制这个节点的遥控器 el 揣在了口袋里。
  • 你写 el.val = 5;，实际上是按了一下遥控器上的按钮，隔空把大仓库里的那个节点的数值改成了 5。

这样做的好处：

因为全靠遥控器隔空操作，Java 在后台雇了一个“清洁工（垃圾回收器）”。一旦发现大仓库里的某个东西没有遥控器连着它了，清洁工就会自动把它收走。你永远不用担心仓库被塞满，也永远不会摸错地方。

2. C++ 的设计思路：绝对自由，怎么爽怎么来

C++ 的核心思想是：“你是老板，你说了算。你想把东西放哪，就放哪。”

C++ 根本不搞“类型决定位置”那一套。不管是一个数字，还是一个巨大的链表，你都可以自由选择存放的方式。在 C++ 的世界里，有三种玩法：

玩法一：直接揣口袋里（值类型）

你写 Node el;。这时候，C++ 直接在你的口袋（栈）里塞进了一个货真价实的节点实体。

  • 它不是遥控器，它就是那个东西本身。
  • 你想改它，直接伸手在口袋里摸到它就能改（el.val = 5）。
  • 当你的函数结束时，你的口袋会被整体翻过来倒干净，这个节点也就跟着一起消失了。不需要任何人来打扫。

玩法二：用“纸条”记录仓库地址（指针）

如果你觉得口袋太小放不下，或者希望这个东西在函数结束后还能留着，你也可以用 new 把它放进大仓库（堆）里。

  • 你写 Node* p = new Node();。这时候，你在仓库里建了一个节点，同时在口袋里放了一张“写着仓库货架号的纸条（指针）”。
  • 你想改它，必须先看一眼纸条上的地址，然后顺着地址找到仓库里的货架（p->val = 5）。

玩法三：给东西起个“外号”（引用）

你嫌天天看纸条找地址太麻烦，C++ 还允许你给已有的东西起个外号：Node& ref = el;。

  • ref 不是新东西，也不是纸条，它就是 el 的另一个名字。打它就等于打 el。

这样做的好处：

速度极快！因为直接把实体放口袋里（栈），省去了去大仓库里来回找货架的时间。但代价是，如果你用纸条（指针）在仓库里建了东西，最后忘了把货架清空（delete），那个东西就会永远占用仓库空间，直到电脑死机。

一种使用struct来定义类的方式，因为没有this指针，构造方式特殊一些，初始化的时候可以使用new但是要用一个指针去接。

3. 一目了然的对比表

我们用最直观的代码和行为来做个总结：

实际场景 Java 的做法 (唯一的路) C++ 的做法 (多条路可选)
创建一个链表节点 必须用 new 建在仓库里，口袋里拿个遥控器。 可以直接塞在口袋里（Node el），也可以写地址放仓库里（Node* p）。
创建一个整数数组 哪怕是 int[]，也必须建在仓库里，拿遥控器控制。 可以直接把整条数组拍在口袋里（int arr[10]）。
把东西传给另一个函数 把口袋里的遥控器（地址）复制一把给对方。两个人控制同一个东西。 默认把口袋里的实体完完整整复制一份给对方（开销大）；除非你主动决定只传纸条（指针）或外号（引用）。
不用的东西怎么处理 遥控器扔了就行，后台清洁工自动打扫。 放在口袋里的会自动消失；放在仓库里的必须自己亲手扔掉，否则仓库会满。

  • Java 就像开全自动挡的普通轿车，所有的机械细节（大仓库、内存地址）都用外壳包起来，不让你碰，只给你一个方向盘和几个遥控按钮。虽然不刺激，但很安全。
  • C++ 就像开手动挡的赛车，引擎、齿轮、轮胎（口袋、仓库、每一个字节的地址）全部赤裸裸地暴露在你面前。你可以玩出最极限的速度，但方向盘要是多打了一点点，车就会当场散架。

论【想当然】

2026-04-13

什么是想当然？

我们生活在一个由数学概率组成的世界里。每件事都有其发生的概率。简单来说，【想当然】就是主观上忽视这件事发生概率比较小

的情况或者主观上对自己不利的情况。这其实并不是一个多么好的习惯，尤其是对于工科生这种以严谨和精确为立身之本的人来说。

想当然的危害

想当然之危害体现在方方面面，这里我从生活和学习两个视角切入。生活上，由于主观上的忽视某个风险，导致对突发事件的准备不

够充分，当某个你自认为的小概率事件真的发生时，往往手忙脚乱，甚至会摧毁你之前做的所有努力。学习中的危害则更加明显，我从我

熟知的编程视角解释，算法题往往都是采用黑盒测试，一个题背后可能有上千个样例点，好的样例会覆盖可能出现的所有情况。在这种情

况下，你认为的小概率事件其实是必然发生的，在这种情况下忽略边界，忽略特殊情况所带来的危害绝对是致命的。

怎么避免想当然

1.坚决避免半场开香槟。用个不算特别恰当的比喻，一定要居安思危。当你觉得最近太顺，在做编程题的时候更明显，当你觉得你做

的很顺，很有可能是你缺乏了一些思考，想当然的把一些潜在的条件忽略掉了。

2.认清想当然的危害，重视重视再重视。千万不要把想当然的危害给弱化掉，觉得无所谓。实际上来看，想当然说是人性的弱点都丝

毫不为过，多少人都会栽在这上面。

3.建立反馈审查机制。很多时候想当然其实是个下意识的动作，很难避免。这时候我们可以多回想回想：我是不是把什么特殊情况忽

略掉了，或者造几个特殊情况的样例，只管的看一下是不是又下意识的犯错了。尤其是在克服 想当然 的初期，多花一些时间，养成习惯

之后就可以很大程度的避免。

4.欲速则不达。多年养成的习惯，甚至于说人性的弱点，某种程度上来说只能缓解不能根除。从日常的小事情做起，一步一步的也无
